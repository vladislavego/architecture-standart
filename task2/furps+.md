# FURPS+ требования (Task2)

Документ описывает архитектурно-значимые требования к MVP цифрового открытия
депозитов в банке «Стандарт».

Требования сформированы с учётом:
- целей бизнеса на этап MVP;
- текущих бизнес-процессов;
- существующего IT-ландшафта банка;
- ограничений текущих систем и подрядчиков.

---

## F — Functional requirements

| ID | Требование | Комментарий |
|----|------------|-------------|
| F1 | На сайте банка отображается список доступных депозитов с актуальными ставками | Сайт используется как маркетинговый канал для привлечения новых клиентов |
| F2 | Клиент может подать заявку на депозит через сайт, указав ФИО и номер телефона | Заявка создаётся как лид, без открытия продукта |
| F3 | Заявка с сайта передаётся в систему кол-центра для обработки | Используется существующая система кол-центра |
| F4 | Менеджер кол-центра связывается с клиентом и консультирует по условиям депозита | В процессе звонка возможны индивидуальные условия |
| F5 | Новый клиент должен пройти очную идентификацию в отделении банка | Регуляторное требование |
| F6 | В интернет-банке клиент видит список доступных депозитов с актуальными ставками | Используется существующий интернет-банк |
| F7 | В интернет-банке отображаются персонализированные ставки по депозитам | Персонализация основана на данных из АБС |
| F8 | Клиент может подать заявку на открытие депозита через интернет-банк | Клиент указывает счёт и сумму депозита |
| F9 | Подача заявки в интернет-банке подтверждается СМС-кодом | Используется существующий СМС-шлюз |
| F10 | На этапе MVP заявка на депозит обрабатывается менеджером бэк-офиса | Полная автоматизация не входит в MVP |
| F11 | Подтверждение условий депозита выполняется в АБС | АБС является системой учёта депозитов |
| F12 | Клиент получает СМС-уведомления о подтверждении ставки и открытии депозита | Уведомления отправляются через СМС-шлюз |

## U — Usability requirements

| ID | Требование | Комментарий |
|----|------------|-------------|
| U1 | Интерфейсы сайта и интернет-банка соответствуют дизайн-системе банка | Используются фирменные цвета и элементы |
| U2 | Процесс подачи заявки должен быть понятным и содержать минимальное количество шагов | Снижение отказов пользователей |
| U3 | Статусы заявки должны быть понятны клиенту | Например: создана, в обработке, подтверждена, депозит открыт |

---

## R — Reliability requirements

| ID | Требование | Комментарий |
|----|------------|-------------|
| R1 | Онлайн-сервисы должны быть доступны 24/7 с целевым SLA не ниже 99,9% | Требование бизнеса |
| R2 | Отказ одного компонента не должен приводить к недоступности всего процесса | Особенно критично для АБС |
| R3 | Архитектура должна минимизировать прямую нагрузку на АБС | АБС масштабируется преимущественно вертикально |
| R4 | В случае сбоя основного ЦОД возможен переход на резервный ЦОД | Актуально для интернет-банка |

---

## P — Performance requirements

| ID | Требование | Комментарий |
|----|------------|-------------|
| P1 | Отклик пользовательских операций должен восприниматься как мгновенный | Ожидание бизнеса |
| P2 | Время загрузки справочных данных не должно превышать 1 секунды | Текущая проблема интернет-банка |
| P3 | Архитектура должна предусматривать кеширование справочных данных | Для снижения нагрузки на БД |

---

## S — Supportability requirements

| ID | Требование | Комментарий |
|----|------------|-------------|
| S1 | Решение должно поддерживать дальнейшую автоматизацию процесса открытия депозитов | Цель финального состояния |
| S2 | Изменения должны минимально затрагивать ядро интернет-банка | Ядро поддерживается подрядчиком |
| S3 | Решение должно быть понятно и сопровождаемо внутренними командами банка | АБС разрабатывается внутри банка |
| S4 | Должна быть подготовлена документация по архитектуре решения | Для дальнейшего развития системы |

---

## + — Additional constraints

| ID | Ограничение | Комментарий |
|----|-------------|-------------|
| +R1 | Использовать существующие технологии банка | .NET Framework, Oracle, Java Spring Boot |
| +R2 | Новые технологии должны быть совместимы с текущими платформами | Минимизация рисков внедрения |
| +R3 | Kafka может использоваться только для новых компонентов | Текущая платформа интернет-банка несовместима с Kafka |
| +R4 | Партнёрский кол-центр не предоставляет API | Возможен только файловый обмен |
| +R5 | Передача чувствительных данных должна быть защищена | Используется шифрование каналов |
| +R6 | АБС остаётся источником истины по депозитным продуктам | Бизнес-логика реализована в PL/SQL |
